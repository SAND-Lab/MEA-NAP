# MEA-Stim Python Port — Plan & Status

Living plan for the MATLAB → Python port of the MEA-NAP **electrical
stimulation detection & analysis** subsystem ("MEA-Stim"). Mirrors the style
of `CATNAP_PORT_PLAN.md` and `PIPELINE_PORT_STATUS.md`. Read this before
touching `src/meanap/stim/`.

## Source of truth: `origin/main`, NOT the checked-out branch

The parity fixtures for this port were generated by MATLAB into
`local/testMEAstim/OutputData20Jul2026/` (dated 20 Jul 2026). That run used the
**MEASTIM_Axion** version of the stim subsystem, merged to `origin/main` via
PR #106 on 2026-07-21. The currently checked-out `feat/plot-parity` branch has
an **older** `Functions/stimAnalysis/` that is missing the shuffle-significance
test and its helpers. **Port against `origin/main`'s versions** of these files
(read them with `git show origin/main:Functions/stimAnalysis/<file>.m`), not the
working-tree copies. Files that exist only on main:
`stimShuffleTest.m`, `plotStimShuffleResults.m`, `computeTrialProportion.m`,
`computeTrialProportionForEachChannel.m`, `getStimArtifactDuration.m`
(plus updated `stimActivityAnalysis.m` / `plotPrePostStimFR.m`).

## Determinism map (what can hit exact parity)

**Deterministic (target exact parity):**
- `detectStimTimes` — pure signal processing on raw voltage (all 6 methods:
  `absPosThreshold`, `absNegThreshold`, `stdNeg`, `blanking`, `longblank`,
  `axionStimEvents`). Refractory pruning. Run-length/blank template logic.
- `getStimPatterns` / `checkStimPattern` — pattern grouping.
- `calculate_psth_metrics` **with `gaussian` smoothing** — this is what the
  **individual-electrode analysis (the CSV metrics) uses** (`psth_bin_width_s=0.001`,
  `gaussian_width_ms=1`). So the headline CSV parity does **not** depend on
  `ssvkernel`. AUC (`trapz`), peak rate/time, half-Rmax, d′, z-score, latency.
- `computeTrialProportion*` → `trialProp_obs`.
- `FiringRateMatrix`, `latencyMatrix`, `ExactSpikeTimes`.
- `getFrAlignedToStim`, `getSpikeLatencyRelStim`, `WithinRanges`,
  `getStimArtifactDuration`, `valuesToColormap`.

**Stochastic (structural/statistical parity only, not bit-exact):**
- `stimShuffleTest` — circular-shift null (no RNG seed).
- Pattern decoding — `fitcecoc` + random node subsets + k-fold CV → sklearn
  equivalent; accuracy comparable in distribution, not bit-exact.
- `ssvkernel` — adaptive-bandwidth KDE (Shimazaki–Shinomoto). Deterministic
  *given* input, but only used for **population-level PSTH plots**, not CSV
  metrics. Port for plot parity; lower priority. (~437-line numeric port.)

## Unit gotchas (from the MATLAB)

- `postStimWindowDur` is in **ms**; most windows in **s**; latencies output in **ms**.
- `artifactDuration = blankDurMode + postStimWindowDur/1000` (s). For
  `longblank`, `blankDurMode = mode(nonStimBlankDur)` (mode of non-stim blank
  durations pooled across channels); 0 for other methods.
- `stimActivityAnalysis` overwrites `rasterBinWidth=0.01` and
  `stimDecodingTimeWindows=linspace(0,0.01,10)` in place — GUI values for those
  are ignored during analysis.
- CSV `channel_id` = electrode ID; `file_index` = electrode position (1..60).
  Only electrodes with a nonzero pattern (i.e. detected stim) get a row.

## Phased ordering (each phase independently validatable)

- **Phase 0 — I/O + fixtures.** Load raw + spikeTimes; write a
  `gen_stim_detection_reference.m` that runs MATLAB `detectStimTimes` on both raw
  files and saves stim times/patterns as fixtures. CSV writer stub.
- **Phase 1 — Detection + patterns.** Port `detection.py`; validate stim
  times/pattern assignment against the Phase-0 fixture (exact).
- **Phase 2 — PSTH + CSV metrics.** Port `psth.py` + `activity.py` (gaussian
  smoothing path); feed MATLAB's own `spikeTimes`; diff every column of
  `StimActivity_NodeLevel.csv` (target exact, like steps 2–4). This is the
  headline deliverable.
- **Phase 3 — Reservoir matrices + shuffle `trialProp_obs`.** Validate
  `FiringRateMatrix`/`latencyMatrix` and observed trial proportion (exact);
  structural checks on the null distribution.
- **Phase 4 — Plots (`ssvkernel`, heatmaps, rasters), decoding, Axion path.**
  Content parity via the plot-parity report tooling.

## Parity strategy (follows the established repo pattern)

- Isolate stim *arithmetic* from spike detection by feeding MATLAB's own
  `spikeTimes` (exactly as `test_pipeline_step2/3/4.py` do).
- Ground-truth fixtures via MATLAB called directly (`gen_*_reference.m` →
  `.npz`), since the pipeline doesn't persist the needed intermediates.
- Stochastic pieces (shuffle null, decoding) get structural/statistical tests,
  not bit-parity — document why in each module docstring.

## Open decisions

- **Stim spike-cleaning propagation into step 2 (undecided).** MATLAB's
  `batchProcessSpikesFromStim` overwrites the spike file at the end of step 1, so
  when `stimulationMode` is on, **step 2's neuronal-activity metrics run on the
  stim-cleaned spikes** (stim-electrode spikes emptied + artifact-window spikes
  removed). This port keeps `clean_spikes_from_stim` isolated inside the stim
  step, so Python's step 2 still uses the raw step-1 spikes. Divergence: mainly
  the stimulating electrode(s) — MATLAB reports 0 firing there, Python reports
  the raw detection (a ~1-electrode effect on blanked data). Options if we fix
  it: (a) rewrite each recording's step-1 `.npz` with cleaned spikes before step
  2 runs (faithful, but mutates step-1 output), or (b) leave step 2 on raw
  spikes (current). **User will decide later** (raised 2026-07-22).

**Known intentional coloring difference — `spikeLatency_*_heatmap`.** Electrodes
with a NaN median latency (no spike in the `(0, 30 ms]` post-stim window across
trials — 21/60 on HUB45) are drawn **light grey** in the port but **dark
purple** in MATLAB. Cause: MATLAB's `valuesToColormap.m` does
`max(0, min(1, normVals))`, and MATLAB's two-argument `min`/`max` silently drop
NaN, so a NaN normalizes to 1 → the last (darkest) colormap entry — painting a
"no-response" electrode identically to the slowest-responding one. The port keeps
grey (distinguishes missing data from max latency) and labels it "no data (N)".
The latency *values* themselves are exact-parity; this is display-only. Decided
with the user 2026-07-22 to keep grey.

### Real bug found & fixed along the way

- **`pipeline/io.py: load_spike_times_mat` ignored MATLAB's empty-array
  marker** (`MATLAB_empty` attr). A zero-spike channel is stored in v7.3 as a
  dataset holding its *shape* (`[0 0]`/`[0 1]`) — the loader read those as two
  spurious spikes, so the stim CSV emitted extra rows for empty channels.
  Fixed via `_read_maybe_empty()`. Latent for the whole pipeline; only bites
  when a channel has zero spikes for a method.

### Parity-critical gotchas (don't re-discover)

- **1-based indexing in detection**: run start sample `i` → time `(i+1)/fs` in
  Python (MATLAB's `i` is 1-based). Wrong here = every stim time off by 1 sample.
- **Only 1 stim electrode** in the test data (the blanked one). The 40/43 CSV
  rows are the *non-stim* channels with spikes in the ±window; the single stim
  electrode (pattern>0) is excluded.
- **CSV metrics use gaussian smoothing** (1 ms), not ssvkernel — the headline
  parity is independent of the KDE port.
- **artifactDuration = blankDurMode + postStimWindowDur/1000** must reach the
  shuffle/reservoir windows; forgetting `blankDurMode` shifts the post-stim
  window start (0.5 ms vs 2.02 ms) and flips a trial's pre/post count.
- **ExperimentMatFiles spikeTimes are raw** (not cleaned); apply
  `clean_spikes_from_stim` (near-no-op on analyzed channels since blanking
  flattens the signal, but empties the stim electrode).
